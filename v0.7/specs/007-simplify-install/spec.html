Feature Specification: インストール簡素化

Feature Branch: 007-simplify-install Created: 2026-02-17 Status: Draft Input: User description: "インストール手順を簡素化して、Homebrewなどのようにインストールスクリプトを実行するだけでセットアップ完了できるようにしたい。..."

User Scenarios & Testing

User Story 1 - ワンライナーでのインストール完了（Priority: P1）

開発初心者のユーザーとして、ターミナルに1行コマンドを貼り付けるだけで wtm コマンドが使える状態になりたい。

Why this priority: インストール体験の根幹であり、これなしには「簡素化」という目標が達成できない。

Independent Test: Unix環境でインストールコマンドを実行したとき、wtm --version が成功することで独立してテストできる。

Acceptance Scenarios:

  1. Given macOS または Linux 環境で、When README.mdに記載された1行のインストールコマンドを実行すると、Then wtm が ~/.local/bin/wtm にインストールされ wtm --version が正常に動作する
  2. Given Windows 環境で、When PowerShell の1行コマンドを実行すると、Then wtm.exe がユーザースコープディレクトリにインストールされ wtm --version が正常に動作する
  3. Given インストールスクリプトが実行されるとき、When プラットフォームとアーキテクチャを自動検出すると、Then 適切なバイナリ（linux-x64, linux-arm, macos-arm64, windows-x64）が自動選択される
  4. Given バイナリダウンロードが完了すると、When SHA256チェックサム検証が自動実行され、Then 検証失敗時はバイナリを削除してインストールが中断される
  5. Given インストール完了後に、When インストール先ディレクトリが PATH に未登録の場合、Then シェル設定ファイルへの追記コマンドが案内される

User Story 2 - README.mdだけでインストール完了（Priority: P1）

プロジェクトを知ったばかりのユーザーとして、GitHub の README.md ページだけを見ればインストール手順が完結したい。

Why this priority: ユーザーが最初に着地するページがREADMEであり、そこからの離脱がツール採用の最大の障壁になっている。

Independent Test: README.mdを読むだけで（外部リンクを踏まずに）インストールコマンドを取得できることで独立してテストできる。

Acceptance Scenarios:

  1. Given ユーザーが kuju63/wt リポジトリのトップページを開くと、When Installation セクションを確認すると、Then コピー&ペーストで実行できるインストールコマンドが表示されている
  2. Given README.mdのInstallationセクションを読むとき、When ユーザーがmacOS/Linux/Windowsのどれを使っていても、Then 各OS向けのコマンドがインライン（外部リンクなし）で確認できる
  3. Given インストールコマンドが記載されているとき、When ユーザーが詳細情報を必要とする場合、Then 詳細情報へのリンクはオプションとして提示され、基本的なインストールには不要である

User Story 3 - ルート権限不要なユーザースコープインストール（Priority: P2）

sudoアクセスを持たないユーザーとして、管理者権限なしに wtm をインストールしたい。

Why this priority: デフォルトの挙動（~/.local/bin へのインストール）によって自然に解決されるが、意識的な設計要件として明示する。

Independent Test: sudo/管理者権限なしにスクリプトを実行し、wtm が利用可能になることで独立してテストできる。

Acceptance Scenarios:

  1. Given デフォルト設定でスクリプトを実行すると、When sudo/管理者権限の要求なしに処理が完了すると、Then バイナリが ~/.local/bin/wtm（Unix）または $env:LOCALAPPDATA\Programs\wtm\wtm.exe（Windows）に配置される
  2. Given システム全体にインストールしたい上級ユーザーが、When --prefix /usr/local オプションを指定すると、Then 指定した場所に配置され（管理者権限が必要な旨を事前案内）
  3. Given インストール先ディレクトリが存在しない場合、When スクリプトが実行されると、Then ディレクトリを自動作成してインストールを続行する

User Story 4 - 再実行で自動アップデート（Priority: P2）

すでに wtm を使用しているユーザーとして、インストールスクリプトを再実行するだけで最新版に更新したい。

Why this priority: 継続的な利用においてアップデートの簡便さも重要だが、新規インストール体験より優先度は低い。

Independent Test: すでに旧バージョンがインストールされている環境でスクリプトを再実行し、最新バージョンに更新されることで独立してテストできる。

Acceptance Scenarios:

  1. Given 旧バージョンの wtm がインストールされているユーザーがターミナル（TTY）でスクリプトを再実行すると、When 新しいバージョンが利用可能な場合、Then 上書き確認プロンプトが表示され、承認後に最新版で上書きして更新が完了する
  2. Given 旧バージョンの wtm がインストールされているユーザーが --force オプションを付けてスクリプトを再実行すると、When 新しいバージョンが利用可能な場合、Then 確認プロンプトなしに既存バイナリを最新版で上書きして更新が完了する
  3. Given 最新バージョンの wtm がすでにインストールされているユーザーがスクリプトを実行すると、When 同一バージョンが検出されると、Then 最新版であることを通知して再インストールをスキップする
  4. Given スクリプトが常に最新バージョンをインストールするとき、When GitHub Releases から最新バージョン情報を自動取得すると、Then ユーザーがバージョンを意識することなく常に最新版が利用できる

User Story 6 - 既存バイナリの上書き確認（Priority: P2）

すでに wtm が存在する環境で初めてインストールするユーザーとして、意図せず既存の wtm を上書きしないよう確認を求めてほしい。

Why this priority: データ損失や予期せぬ上書きを防ぐ安全策として重要。また --force による確認スキップは CI/CD 等の自動化環境でも必要。

Independent Test: インストール先に既存バイナリが存在する環境でスクリプトを実行し、インタラクティブ時は確認プロンプトが、--force 指定時はプロンプトなしで上書きが完了することで独立してテストできる。

Acceptance Scenarios:

  1. Given インストール先に既存の wtm バイナリが存在するとき、When ターミナル（TTY）でスクリプトを実行すると、Then 上書き確認プロンプト（y/N）が表示され、y を入力した場合のみ上書きが実行される
  2. Given インストール先に既存の wtm バイナリが存在するとき、When 確認プロンプトで n または Enter（デフォルト拒否）を入力すると、Then インストールがキャンセルされ、既存バイナリは変更されない
  3. Given インストール先に既存の wtm バイナリが存在するとき、When --force オプションを付けてスクリプトを実行すると、Then 確認プロンプトなしに上書きが完了する
  4. Given インストールスクリプトがパイプ経由（curl | sh 等）で実行され非インタラクティブ環境の場合、When インストール先に既存の wtm バイナリが存在すると、Then プロンプトを表示せず上書きをスキップし、--force フラグの使用方法を案内するメッセージが表示される

User Story 5 - 失敗時の分かりやすいエラーガイダンス（Priority: P3）

インストールに失敗したユーザーとして、問題の原因と解決方法を即座に把握したい。

Why this priority: 異常系の体験品質を高めるが、正常系が動作することが先決。

Independent Test: ネットワーク不通・非対応プラットフォームなどの障害条件でスクリプトを実行し、理解しやすいエラーメッセージが表示されることで独立してテストできる。

Acceptance Scenarios:

  1. Given ネットワーク接続がない環境でスクリプトを実行すると、When ダウンロードに失敗すると、Then ネットワーク接続確認を促すメッセージが表示される
  2. Given 非対応プラットフォームでスクリプトを実行すると、When プラットフォーム検出が失敗すると、Then 対応プラットフォーム一覧と手動インストールへの案内が表示される
  3. Given SHA256検証に失敗すると、When チェックサム不一致が検出されると、Then ダウンロードファイルを削除し再試行を促すメッセージが表示される

Edge Cases

  • インストール先に既存の wtm バイナリが存在する場合（初回インストール・アップデート共通）: TTY環境では y/N 確認プロンプトを表示（デフォルト拒否）する。--force フラグ指定時は確認をスキップして上書きを実行する。非インタラクティブ環境（パイプ実行等）では上書きをスキップし、--force フラグの使用方法を案内するメッセージを表示する（FR-016/FR-017/FR-018 参照）
  • curl も wget も存在しない環境: 両方試みた後に手動ダウンロードの手順を案内する
  • GitHub Releases API レート制限: 最新バージョン情報の取得に失敗した場合、しばらく時間をおいて再試行するよう案内するメッセージを表示する
  • macOS Gatekeeper によるブロック: インストール後に Gatekeeper 解除手順を出力する
  • 確認プロンプトでタイムアウト（標準入力が閉じられた場合）: デフォルト動作（拒否）を適用してインストールをキャンセルする

Requirements

Functional Requirements

  • FR-001: インストールスクリプト（Unix向け .sh / Windows向け .ps1）をリリース資産に含める
  • FR-002: スクリプトはプラットフォームとアーキテクチャを自動検出し対応バイナリを自動選択する
  • FR-003: デフォルトインストール先はルート権限不要なユーザースコープディレクトリとする（Unix: ~/.local/bin、Windows: $env:LOCALAPPDATA\Programs\wtm）
  • FR-004: バイナリダウンロード後にSHA256チェックサム検証を自動実行し、失敗時はバイナリを削除してインストールを中断する
  • FR-005: Unix環境でバイナリに実行権限を自動付与する（ユーザーによるchmod操作を不要にする）
  • FR-006: インストール先ディレクトリが存在しない場合、自動作成する
  • FR-007: インストール先が PATH 未登録の場合、シェル設定ファイルへの PATH 追記コマンドを出力する
  • FR-008: スクリプトは常に最新バージョンを自動取得してインストールする（バージョン指定は提供しない）
  • FR-009: 再実行時に既存インストールより新しいバージョンが存在する場合、インストールフローを経てアップデートする（TTY環境では FR-016 に従い上書き確認プロンプトを表示する。--force 指定時または非インタラクティブ環境では FR-017/FR-018 に従う）
  • FR-010: 再実行時に同一バージョンが既にインストール済みの場合、最新版である旨を通知してスキップする
  • FR-011: --prefix <DIR> オプションでインストール先を指定できる
  • FR-012: README.md の Installation セクションは、外部リンクなしに各OS向けのインストールコマンドをインライン表示する
  • FR-013: インストールスクリプトはバージョン番号を含まない安定した永続URLから配布される
  • FR-014: リリースワークフローはインストールスクリプトをリリース資産として自動含める
  • FR-015: すべてのエラーケースで、問題内容と解決策を示す分かりやすいメッセージを出力して終了する
  • FR-016: インストール先に既存バイナリが存在し、かつインタラクティブ環境（TTY）で実行された場合、上書き確認プロンプト（y/N、デフォルト拒否）を表示する。ユーザーが拒否した場合はインストールをキャンセルし、既存バイナリを変更しない
  • FR-017: --force フラグを指定した場合、既存バイナリへの上書き確認プロンプトをスキップして上書きを実行する
  • FR-018: 非インタラクティブ環境（パイプ経由実行、標準入力が TTY でない場合）では確認プロンプトを表示せず、上書きをスキップし --force フラグの使用方法を案内するメッセージを出力する

Key Entities

  • インストールスクリプト（Unix）: curl | sh または wget | sh パターンで実行可能な .sh ファイル。プラットフォーム検出・ダウンロード・検証・配置・権限設定・PATH案内を一括担当
  • インストールスクリプト（Windows）: PowerShellで実行可能な .ps1 ファイル。Unix版と対称的な機能セット
  • 安定インストールURL: バージョン非依存の永続URL（例: https://kuju63.github.io/wt/install.sh）

Success Criteria

Measurable Outcomes

  • SC-001: インストールコマンド1行の実行から wtm --version 確認までの所要時間が、手動インストール手順と比べて80%以上削減される（想定: 手動5分以上 → スクリプト1分以内）
  • SC-002: README.mdのInstallationセクションを読んだユーザーが外部ページ遷移なしにインストールコマンドを取得できる率が100%
  • SC-003: Unix/Windowsの全対応プラットフォームで、1行コマンドのみでルート権限なしにインストールが完了する
  • SC-004: SHA256検証を含む全ステップがユーザーの手動操作なしに完了する
  • SC-005: スクリプト配布URLは将来バージョンアップ後も変更されない（バージョン非依存の安定URL）
  • SC-006: インストールスクリプトの再実行だけで常に最新版に更新でき、ユーザーがバージョンを意識する必要がない
  • SC-007: 既存の手動インストール手順は引き続き機能し、既存ユーザーへの影響ゼロ
  • SC-008: スクリプト失敗時、ユーザーが出力メッセージを読むだけで次のアクションを把握できる（外部ドキュメント参照を必須としない）

Assumptions

  • インストールスクリプトは GitHub Releases に加え、GitHub Pages 経由でも安定URLとして提供されることを想定
  • Unix向けは curl または wget のいずれかが利用可能であることを前提
  • Windows向けは PowerShell 5.1 以上が利用可能であることを前提（Windows 10以降では標準搭載）
  • 既存の手動インストール手順は変更しない（本機能は追加の手段）
  • リリースバイナリのファイル名は wtm（Unix系）および wtm.exe（Windows）とする

Out of Scope

  • Homebrew Tap、Scoop、apt/yumリポジトリへの公式対応（将来の拡張候補）
  • NuGetの dotnet tool install への対応（別途フィーチャー）
  • GUI インストーラー（.msi、.pkgなど）の提供
  • ARM版Windowsバイナリのインストール対応
  • インストールスクリプト自体のGPG署名検証（将来のセキュリティ強化候補）
  • アンインストールスクリプトの提供（別途フィーチャー）

Dependencies

  • 006-release-hash-files で実装されたSHA256ハッシュファイル生成機能（SHA256SUMS、個別.sha256ファイル）に依存
  • GitHub Pages デプロイ設定（既存のbuild-and-deploy-docsジョブへの統合を想定）
  • リリースワークフロー（.github/workflows/release.yml）へのスクリプト追加が必要
